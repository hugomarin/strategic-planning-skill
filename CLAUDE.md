# Deus — Project Context

This project uses the `strategic-planning` v6.0 (Phase 1) and `technical-context` v1.0 (Phase 2) with a multi-agent architecture.

## What this project is

Deus is a Product Harness for converting strategic intention into verifiable, versioned production artifacts. This folder covers Phase 1 (Strategic Planning) and Phase 2 (Implementation Context).

## Skill versions

- `strategic-planning` v6.0 — active. Location: `.claude/skills/strategic-planning/SKILL.md`
- `technical-context` v1.0 — active. Location: `.claude/skills/technical-context/SKILL.md`

Each skill is fully self-contained — no dependency on globally installed plugins.

## Command dispatch

### Phase 1 — `/sp-*` commands

When the user sends any message beginning with `/sp-` or `sp-`, read `.claude/skills/strategic-planning/SKILL.md` before executing. Do not execute from memory.

### Phase 2 — `/impl-*` commands

When the user sends any message beginning with `/impl-` or `impl-`, read `.claude/skills/technical-context/SKILL.md` before executing. Do not execute from memory.

### Version validation

Before executing any `/sp-*` command, verify that the active skill version matches the version declared in the **Skill versions** section above.

1. Read the `version:` field from the frontmatter of the active skill file.
2. Compare against the version declared here.
3. If they match → proceed normally.
4. If they do not match → stop and notify: "CLAUDE.md declares skill vX.X but the active skill file is vY.Y. Run `/sp-sync` to synchronize before proceeding."

Do not proceed with a mismatched version without explicit user confirmation.

## Agent architecture

Two shared subagents operate across both Phases:

- **Diagnostic Analyst** — `.claude/agents/diagnostic-analyst.md`. Supports both `strategic` and `implementation` scopes. Always invoked with a `scope` parameter. Returns structured diagnostic report.
- **Adversarial Reviewer** — `.claude/agents/adversarial-reviewer.md`. Supports `strategic`, `roadmap`, and `implementation` scopes. Always invoked with a `review_scope` parameter. Returns two-block review report.

Each Orchestrator (skill) decides, writes, versions, and publishes. Subagents only analyze and report — they never write files or make decisions.

Review report schema: `.claude/references/review-report-format.md`

## Folder conventions

### Input/
- `Input/Strategic/` — source docs for Phase 1. Read-only.
- `Input/Implementation/` — source docs for Phase 2. Read-only.

### Phase 1 output
- `Strategic Artifacts/` — all KAs generated by Phase 1 live here.
- `_analysis.md` — temporary scaffolding inside `Strategic Artifacts/`. Deleted before handoff.

### Phase 2 output
- `Technical Artifacts/` — all implementation artifacts live here: `stack.md`, `schema.md`, `architecture.md`, `features/`.
- `_technical_analysis.md` — temporary scaffolding inside `Technical Artifacts/`. Deleted before handoff.

### Logs (separate per Phase)
- `Strategic Artifacts/strategic-change-log.md` — decisions log for Phase 1.
- `Technical Artifacts/technical-change-log.md` — decisions log for Phase 2.
- `Strategic Artifacts/open-questions.md` — open questions for Phase 1.
- `Technical Artifacts/technical-open-questions.md` — open questions for Phase 2.

### Skill files
- `.claude/agents/` — shared subagent definitions (multi-scope).
- `.claude/references/` — shared schemas and formats.
- `.claude/skills/strategic-planning/` — Phase 1 skill + references.
- `.claude/skills/technical-context/` — Phase 2 skill + references.
- `.claude/commands/` — all registered commands for both Phases.

### Strategic context available to Phase 2
`technical-context` may read `Strategic Artifacts/` for strategic context (product thesis, user architecture, product architecture). It never modifies these files.
